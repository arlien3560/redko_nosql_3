## Task 1
## Схема графа (MovieLens)

```
┌─────────────────────────────┐                 ┌──────────────────────────┐
│            User             │                 │          Movie           │
├─────────────────────────────┤    RATED        ├──────────────────────────┤
│  userId      : Integer  (PK)│  {rating,       │  movieId   : Integer (PK)│
│  gender      : String       │   timestamp}    │  title     : String      │
│  age         : Integer      │ ──────────────▶ │  year      : Integer     │
│  occupation  : Integer      │                 │                          │
└─────────────────────────────┘                 └────────────┬─────────────┘
                                                              │
                                                              │ HAS_GENRE
                                                              │
                                                              ▼
                                                 ┌──────────────────────────┐
                                                 │          Genre           │
                                                 ├──────────────────────────┤
                                                 │  name      : String  (PK)│
                                                 └──────────────────────────┘
```

### Вузли (nodes)

| Лейбл   | Властивості                              | Ключ              |
|---------|------------------------------------------|-------------------|
| `User`  | `userId`, `gender`, `age`, `occupation`  | `userId` (синт.)  |
| `Movie` | `movieId`, `title`, `year`               | `movieId` (синт.) |
| `Genre` | `name`                                   | `name` (натур.)   |

### Ребра (relationships)

| Тип          | Напрямок              | Властивості           |
|--------------|-----------------------|-----------------------|
| `RATED`      | `(User) ──▶ (Movie)`  | `rating`, `timestamp` |
| `HAS_GENRE`  | `(Movie) ──▶ (Genre)` | —                     |

### Constraints (унікальність + індекс)

```cypher
CREATE CONSTRAINT user_id_unique FOR (u:User)  REQUIRE u.userId  IS UNIQUE;
CREATE CONSTRAINT movie_id_unique FOR (m:Movie) REQUIRE m.movieId IS UNIQUE;
CREATE CONSTRAINT genre_unique FOR (g:Genre) REQUIRE g.name IS UNIQUE;
```

1. Вузлами стали User, Movie, Genre, бо це самостійні сутності, які мають власну ідентичність і існують незалежно. 
Ребрами стали RATED та HAS_GENRE, бо вони виражають зв'язок між сутностями і не мають сенсу у відриві від пари вузлів, які вони з'єднують.
Логіка визначення ключів - синтетичні та натуральні виходить з природи опису даних. Оскільки автори бази даних чітко вказують, що дані сильно розріджені для User та Movie, то вони потребують окремих синтетичних ключів.

2. Оцінка стала ребром, бо вона відображає зв'язок між фільмом та користувачем і не існує у відриві від цієї пари. Оцінка природно живе між користувачем і фільмом.
Компроміс цього підходу пов'язаний з супервузлами: в популярних фільмах з'являться багато зв'язків і якщо ми робитимемо запити, які будуть потребувати багато обходів, то це може спричинити ускалдення запиту (підвищення розходу пам'яті та slow query) - тож потрібно свідомо обмежувати запити на ширину і глибину обходів.

3. Згідно до README архіву, жанрів в списку всього 18, назви унікальні та стабільні. Це дозволяє в ряді запитів збирати їх в єдиній точці. Якщо б ми зберігали їх лише як властивість, то це ризик Full scan - а цього треба уникати, особливо при великих масивах даних

# Task 3

1. Довжина шляху - це скільки раз треба пройти ребро, щоб дістатись від одного користувача до іншого.
2. Hop - це 1 крок по ребру. Тобто в нашому прикладі логіка така: між одним користувачем і іншим нас розділяє фильм, який він оцінив, тож в нашому випадку маємо мінімум 2 hops, щоб поєднати користувачів через нього.
3. Шлях довжини 4 - це значить, що перехід відбувся User1 -> Movie1 -> User_Middleware -> Movie2 -> User2. Тобто ми прийшли до результату через Користувача-Посередника. Якщо шлях довжини збільшити до 6, то в нас з'явиться за аналогією ще 1 Користувач-Посередник. Простіше кажучи, конкретно в нашому прикладі дводольного зв'язку в графі 2 hops додають 1 Користувача в глибину пошуку.
